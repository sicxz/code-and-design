CSS Scroll-Driven API

Animations the
browser controls.

Modern CSS can link an animation directly to scroll position. No JavaScript. No event listeners. The browser handles it — off the main thread — for smooth, declarative scroll effects.

The shift

From JavaScript
to CSS declaration.

Old approach: A JavaScript scroll event fires many times per second. It reads the scroll position, calculates a new CSS value, and applies it. All of that runs on the main browser thread — the same thread handling user input.

New approach: You declare the relationship in CSS. The browser maps scroll position to animation progress internally, on the compositor thread, without JavaScript ever being called.

The result: smoother animations, simpler code, no layout thrashing.

The one-sentence model

Scroll distance replaces time as the thing that drives an animation forward. Instead of "this animation runs for 1 second," you're saying "this animation runs over 500px of scroll."

⚠
Browser support note: CSS scroll-driven animations are supported in Chrome 115+, Edge 115+, and Firefox 110+ (behind a flag). Safari support is in progress. For production work requiring broad support, use GSAP. Check caniuse.com for current status.

The two timeline types

Scroll Progress vs.
View Progress.

These are the two ways to define what drives an animation. They answer different questions.

Scroll Progress Timeline

How far has the container scrolled?

The timeline runs from 0% (top of scroll container) to 100% (bottom). The animation progresses as the user scrolls the entire page.

Use this for: reading progress bars, parallax backgrounds, elements that animate across the full page scroll.

animation-timeline: scroll();
View Progress Timeline

How far has this element traveled through view?

The timeline starts when the element enters the viewport and ends when it leaves. Each element has its own timeline — based on its own visibility.

Use this for: reveal-on-scroll, per-element entrance animations, triggering when content is in view.

animation-timeline: view();
Quick rule of thumb

If the animation should reflect how far the page has scrolled → use scroll(). If it should reflect where a specific element is on screen → use view().

The CSS Properties

Four properties.
One idea each.

animation-timeline
The core property. Tells an animation to use scroll progress instead of a clock. Accepts scroll(), view(), or a named timeline. This is the one that unlocks everything else.
scroll-timeline-name
Applied to a scroll container. Gives the container a name (must start with --) so other elements can reference it. Essential for parallax effects where multiple elements animate from the same scroll source.
view-timeline-name
Applied to an element. Names a view-based timeline so other elements can reference this element's scroll visibility. Useful when an element's position in view should drive an animation somewhere else on the page.
animation-range
Only works with view() timelines. Controls when the animation starts and ends relative to the element entering and exiting the viewport. Format: "[keyword] [%] [keyword] [%]"
Required but irrelevant

When using animation-timeline, the spec still requires animation-duration to be set. The value doesn't matter — scroll position is now in control, not time. Set it to 1ms and forget it.

The code

Three patterns you'll
use constantly.

Pattern 1 — Reading progress bar (scroll)

The entire page scroll drives a progress bar from 0 to full width. This is what's running at the top of this page.

/* 1. Define what the animation does */ @keyframes grow-bar { from { transform: scaleX(0); } to { transform: scaleX(1); } } /* 2. Apply it and link to scroll */ #progress-bar { transform-origin: left center; animation-name: grow-bar; animation-timeline: scroll(root); /* page scroll */ animation-duration: 1ms; /* required, ignored */ animation-fill-mode: both; }
scroll(root)Watches the root scroll container — the page itself. Use scroll() with no args for the nearest scrollable ancestor.
transform: scaleX()Always animate with transform and opacity when possible. These run on the GPU and don't trigger layout recalculation.
animation-fill-mode: bothKeeps the animated state before and after. Without this, the element snaps back when scroll is at 0%.

Pattern 2 — Reveal on scroll (view)

An element fades in as it enters the viewport. The scroll section of this page uses this pattern — watch the cards below as they scroll into view.

@keyframes reveal { from { opacity: 0; transform: translateY(30px); } to { opacity: 1; transform: translateY(0); } } .card { animation: reveal linear both; animation-timeline: view(); /* this element's visibility */ animation-range: entry 0% cover 40%; /* start entering → 40% covered */ animation-duration: 1ms; }
view()Creates a timeline based on this specific element's position in the viewport. Each card gets its own independent timeline.
animation-rangeControls precisely when the animation starts and ends. "entry 0%" = element begins entering. "cover 40%" = 40% of the way through the viewport.
linearThe easing for the animation itself. With scroll control, 'linear' feels natural. Time-based easing is less important here.

Pattern 3 — Named timeline for parallax

Multiple elements animate in response to the same scroll container. You name the timeline on the container and reference it by name.

/* Container defines and names the timeline */ .scroll-section { scroll-timeline-name: --section-scroll; scroll-timeline-axis: block; /* vertical */ overflow-y: scroll; height: 400px; } /* Children reference the named timeline */ .bg-layer { animation-name: drift-slow; animation-timeline: --section-scroll; /* by name */ animation-duration: 1ms; } .fg-layer { animation-name: drift-fast; animation-timeline: --section-scroll; /* same source */ animation-duration: 1ms; }
--section-scrollNamed timelines start with double dash, like CSS custom properties. They're scoped to the element and its descendants.
Both layers, same sourceBoth fg-layer and bg-layer respond to the same scroll container's progress — but with different keyframe values, creating the speed differential that produces depth.
When to use namedWhen multiple elements need to react to the same scroll source — especially useful for parallax depth effects inside a bounded container.

Deep dive — animation-range

Precise control over
when things happen.

animation-range only applies to view() timelines. It defines when in the element's viewport journey the animation starts and ends.

THE ELEMENT'S JOURNEY THROUGH THE VIEWPORT

entry 0%
entry 100%
cover 50%
exit 0%
exit 100%
entry 0% cover 50%
Starts when the element first peeks into the viewport. Finishes when it's 50% visible. Classic "reveal" timing.
cover 0% exit 100%
Starts when the element fully enters the viewport. Runs through until it exits. Good for sustained effects.

Type · scroll()

Reading Progress Timeline

Tracks the overall scroll position of the page from top to bottom. Used for progress bars, background parallax, anything that spans the whole page.

Type · view()

View Progress Timeline

Tracks an element's visibility in the viewport. Each element gets its own timeline. Perfect for reveal-on-scroll animations and entrance effects.

Type · Named timeline

Named Scroll Timeline

A timeline defined on a scroll container and named with a CSS custom property. Child elements reference it by name — essential for multi-element parallax.

/* The CSS powering these exact cards */ .vdc { animation: card-reveal linear both; animation-timeline: view(); animation-range: entry 0% cover 35%; animation-duration: 1ms; } @keyframes card-reveal { from { opacity: 0; transform: translateY(30px); } to { opacity: 1; transform: translateY(0); } }

Knowing the limit

When CSS isn't enough.

CSS scroll-driven animations handle many patterns well. But there are things they can't do — and that's where GSAP comes in.
