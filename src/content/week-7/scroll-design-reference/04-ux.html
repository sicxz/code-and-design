04

UX + Accessibility

Motion that harms
is never a design choice.

The technical capability to animate doesn't mean you should. Scroll-driven animation carries real responsibility: vestibular disorders, performance costs, narrative interference. Here's how to handle all three.

The real risk

Some users can't watch
your animations.

35%
of adults over 40
have vestibular dysfunction

Vestibular mismatch — what it is.

Your eyes perceive motion. Your inner ear doesn't register it. The sensory conflict can cause dizziness, nausea, headaches — the same physiological response as motion sickness in a car. Large parallax effects, rapid movement, and scroll-hijacking are the most common triggers.

Millions of people enable "Reduce Motion" in their system settings specifically because of this. When you respect that setting, you're not reducing your design — you're making it safe for everyone.

The number that matters

The Web Accessibility Initiative estimates vestibular disorders affect 35% of adults 40+. That's not a niche edge case. It's likely 20–30% of your actual audience.

The solution

One media query.
Full responsibility.

The browser can tell you whether the user has requested reduced motion. prefers-reduced-motion: reduce is set when the user has enabled "Reduce Motion" in their operating system preferences.

Live demo — prefers-reduced-motion Motion: normal Reduced motion detected — animation disabled

The box above animates if your system motion preference is normal. If you've enabled "Reduce Motion" in System Settings, the animation is already disabled — that's the media query working. To test: on Mac, go to System Settings → Accessibility → Display → Reduce Motion.

In CSS

/* Default: animations are on */ .animated-element { animation: slide-in linear; animation-timeline: view(); animation-duration: 1ms; } /* Fallback: disable for users who need it */ @media (prefers-reduced-motion: reduce) { .animated-element { animation: none; /* element appears in its final state */ } }
Strategy A — disableThe safest option. The element appears in its final visible state immediately. No motion at all. Suitable for most scroll reveal effects.
Strategy B — reduceReplace the large, fast animation with something minimal — a simple fade with no position change, for example. Keeps visual feedback without vestibular risk.
animation: noneThis cancels the animation entirely. The fill-mode ensures the element stays visible at its target state rather than disappearing.

In GSAP

// GSAP's matchMedia lets you define different behavior for different conditions ScrollTrigger.matchMedia({ // For users with normal motion preference "(prefers-reduced-motion: no-preference)": function() { // Set up all your complex GSAP animations here gsap.timeline({ scrollTrigger: { trigger: ".scene", pin: true, scrub: 1 } }) .to(".element", { x: 400, rotation: 360 }); }, // For users who prefer reduced motion — simpler or no animation "(prefers-reduced-motion: reduce)": function() { // Either skip animations, or use simple fades only gsap.set(".element", { opacity: 1 }); // just make it visible } });

Performance

Animate the right
properties.

Not all CSS properties perform equally when animated. Some trigger the browser to recalculate layout, repaint the screen, and composite layers. Others skip most of that work.

✓ Animate these — GPU handled
transform: translateX()
transform: translateY()
transform: scale()
transform: rotate()
opacity
filter (with caution)
These run on the compositor thread, separate from the main browser thread. They can't trigger layout recalculation — which is why they're smooth.
✗ Avoid these — trigger layout
width / height
top / left / right / bottom
margin / padding
font-size
background-position (old trick)
These force the browser to recalculate layout for potentially the entire page on every animation frame. This is the source of "jank" — the stuttering you see on complex pages.
will-change — use sparingly

Adding will-change: transform to an animated element hints to the browser to promote it to its own GPU layer. This can smooth animations — but applying it to many elements wastes memory. Only use it on elements you know will animate, and remove it after the animation completes if possible.

The line not to cross

Never hijack
the scroll.

Scroll hijacking means taking control of the scrollbar — changing its speed, locking it, or making it move at an unnatural rate during animation. This is almost universally bad UX.

It breaks the user's expectation of how their own input device works. It can trap users who are trying to navigate quickly or skim content. And it's a significant accessibility failure.

Scroll linking — the correct approach — connects animation progress to the natural scroll position without altering the scroll itself. The user scrolls at their own pace; the animation responds accordingly.

Scroll hijacking (avoid)

JavaScript manually controls scroll position. User scrolls one "click" but the page jumps to a defined section. The scrollbar doesn't reflect where the user actually is. Users lose the ability to skim.

Scroll linking (use this)

Animation progress maps to scroll position, but scroll position is never overridden. User scrolls at their speed. The animation plays proportionally. Content is always accessible by skipping past.

The judgment call

When to use it.
When not to.

Use scrollytelling when...
The narrative is linear and sequential — one thing leads to the next.
Visuals are essential to understanding — the animation explains, not decorates.
You want to control the pace of information delivery deliberately.
The content is long-form: an article, a data story, a product walkthrough.
Creating immersive, brand-forward experiences where engagement is the goal.
Avoid scrollytelling when...
Users need to scan or find specific information quickly.
The page is primarily transactional: forms, checkout, sign-up.
Content hierarchy is complex and non-linear — users need to jump around.
Performance on low-end devices is a primary concern.
You can't provide a working reduced-motion fallback.
